# 社区邀请规范（cue&skill 用户群）

`cue-buddy` 与 `cue-research` 共用本规范。「Cue&Skill 用户群」的价值 = **用户使用答疑 + 最新搭子模板分享**。在**高意图时刻**邀请用户加入——克制、不刷屏、不每次都弹。

> 用户面前只讲"答疑 + 拿最新模板"的真实价值；"推广/催更"是平台侧目的，**不写进给用户的邀请文案**。

## ⚠️ 适配边界（发布前必读）

当前群是**内部群**（`external=False`，单租户）：**只有同租户用户能扫码/点链接加入**；**外部租户用户（其它公司用开源 skill 的人）无法加入**（飞书：外部联系人不能加入内部群）。因此：

- **若你的用户都在同一飞书租户（内部）** → 直接用，链路通畅（`membership_approval=no_approval_required`，扫码即进）。
- **若用户可能是外部** → 需改用**外部可加入的群**（外部群 / 允许外部加入），否则**不要**对外部上下文展示二维码（会"看得到进不去"）。在确认前，把邀请当"内部用户"功能用。

## 何时呈现（4 触发 + 冷却）

核心：**只在高意图时刻推，永远一行轻量附加，不打断主任务。**

| 触发 | 时机 | 呈现 | 冷却 |
|---|---|---|---|
| ① 首次使用 | onboarding 首屏，**一行文字 + 链接** | 不展示大二维码 | 一次性（全局，跨两个 skill 共用 `onboarded`） |
| ② 完成有价值动作后 | `+create` 成功 / 跑完一次 research **交付之后**（report + 满意度/下一步问题**都给完**，再附一行；**不得插在交付中间**） | 一行文字 + 链接 | 14 天一次 |
| ③ 卡住 / 报错 | **先**帮用户处理/解释/给下一步，**再**把群作为**温和兜底**（"还卡着的话群里也能问"）——**不是报错就把用户甩去群里** | 一行文字 + 链接 | 14 天一次 |
| ④ 用户显式问 | "怎么加群 / 社区 / 反馈 / 有没有新模板" | **展示二维码图片**（支持图片的端）+ 链接 | **不冷却，随叫随给** |

> **二维码只在 ④（用户主动要）时展示**；①②③ 被动触发只给一行文字 + 链接，避免大图破坏"一行附加"。纯文本端一律给链接；用户要二维码可 `lark-cli auth qrcode <链接> --ascii` 现生成。

## 冷却机制

时间戳文件 `~/.cue/last-community-invite.json`：`{"onboarded": true, "last_passive_shown": <epoch_seconds>}`。

- **被动触发（②③）前先读**：距 `last_passive_shown` < 14 天（1209600s）则**跳过**；展示后写回当前时间戳（必要时先 `mkdir -p ~/.cue`）。
- **每会话被动邀请最多一次**——这是**首要约束**：即使文件读/写失败，也**按"本会话已展示"处理，本会话内不再弹**（失败时**默认不弹**被动邀请，宁可漏不可扰），不报错打断主任务。
- **① 首次**：`onboarded` 为假才显示，显示后置 `true`（全局一次：用户在任一 skill onboarding 过，另一个不再重复）。
- **④ 用户显式问**：永远响应，**不受冷却**、不更新被动时间戳。

## 呈现规则（别变成 spam）

- 邀请永远是**主任务之后的一行附加**，不抢戏、不刷屏、不打断交付闭环。
- 文案随场景微调，但都落"答疑 + 最新模板"的真实价值；**不写"求关注/求转发/催更"式话术、不承诺收益、不夸大**。
- 群是**公开交流**性质但**链接永久**：运营侧需有管理员 + 基本审核（防滥用扩散）；功能侧不因此夸大。
