# Hard Rules — 模板铁律

R1-R7 + R9 均有 `+validate` 自动检测,违反 `error` 级会拒绝 `+create / +update`,`warning` 级允许通过但强烈建议改。R8 不是铁律,是设计经验法则——见末尾 [设计经验法则](#design-heuristics) 章节,跟 R1-R9 hard rules 分开,**不**走 validator。

本文是 `~/work/buddy/skills/developing-buddies/SKILL.md` 的公开发布版（脱去内部基础设施引用）。

---

## 总原则

1. **搭子是证据收集器，不是决策者** — 不替用户下"建议进入/暂缓进入"的最终判断
2. **prose 不出现工具名** — 工具治理是后端约束，前端 LLM 文本不应耦合
3. **按数据源聚类调研** — 不按章节顺序线性走
4. **变量串通三处** — input_form_spec / 主标题 / 关键配置 三处变量必须完全一致
5. **声明边界** — 明确说什么不做（不替代正式尽调/合规审查/法律意见/真正反洗钱 AML/医疗诊断 等）

## R1. input_form_spec 必须三段式变量 [error]

### 规则
`input_form_spec` 必须满足：
- 单行（无换行）
- 以 `需提供:` 或 `需提供：` 开头
- 至少 1 个三段式变量 `[属性_主体_类型]`（中文，下划线分段）
- 含 `可提供:` 段（可选输入）
- 建议含 `(默认: ...)` 兜底（warning 级）

### 为什么
- 前端把 `[...]` 渲染成可编辑输入框；非三段式 → 无法识别
- 单行 → 表单只渲染一行 helper 文字
- 默认值 → 用户没填可选项时也能跑

### 反例
- `请提供企业名和分析重点` — 无三段式变量
- `需提供:\n[目标_授信_企业]` — 多行
- `需提供: [企业]` — 单段不是三段式

## R2. goal 必须简洁有力、价值优先的一段简介 [error/warning]

> goal **就是搭子在 playbook 卡片上的简介**——讲**解决什么问题 / 给什么价值**，
> 不讲"怎么实现"。它既是卡片文案，也是给 researcher 的目标陈述。

### 规则
- 必须是一整段，不写编号清单 `1. ... 2. ...`（error）
- **简洁有力**：建议 ~40-80 字；>100 字 warning、>200 字 error（怎么做的维度细节放 search_plan，不堆进 goal）
- **价值优先**：以"解决什么问题/给什么价值"开头；**不要**用"作为…(长角色)…助手，围绕…，[一长串怎么做]"这类运行手册式前缀（>90 字的长角色前缀 warning）
- **不泄漏实现/技术词**：goal 里不得出现 `LangGraph/循环图/批处理/MECE/pipeline/状态机/工作流/评估模型/估值模型/自适应/向量/大模型/prompt` 等内部实现表述（error）
- **不硬码主体**：用占位变量或泛指，别把某个示范主体写进角色（卡片会把 `[属性_主体_类型]` 泛化显示；硬码会出现"作为 中信证券 …助手"这种错位）
- **不写免责声明**：`不替代/不构成/仅做…/严格不…/凡公开未披露…` 等边界声明放 search_plan 避坑指南或 report_format（warning）
- 不出现"建议进入/谨慎进入/暂缓进入/建议授信/拒绝授信"等决策语

### 为什么
- goal 同时是卡片文案：冗长、堆"怎么做"、泄漏实现、硬码主体，都会让用户看不明白搭子的价值（2026-05-27 全量重写 50 个 goal 的教训）
- 编号清单让 LLM 把任务当机械流程而非角色代入 → 报告生硬
- 决策语越界，让搭子去做需要专业审批的判断 → 法律/合规风险

### 正例 / 反例
```
✅ 横向比对各主要券商的两融政策（折算率、标的、保证金），识别竞对抢客与风险收缩信号、研判本方杠杆吸引力强弱，产出晨会可直接用的竞对情报。  (73字, 价值优先)
❌ 作为 [本方_券商_主体] 融资融券业务与客户经理的竞对情报助手，围绕各主要券商在担保品折算率、融资融券标的覆盖…[200+字怎么做]…；不提供个股买卖建议，不替代…  (运行手册式, 冗长)
❌ 作为政企数据尽调专家，基于 LangGraph 循环图计算特性，为您构建多源穿透+批处理…  (泄漏实现)

### 反例
```
1. 提供基本信息
2. 提供财务数据
3. 给授信建议
```

## R2b. title 简洁有力、体现价值 [warning]

### 规则
- `title` 是搭子在卡片上的**名字**：简洁有力、一眼看出价值，建议 ~≤8-10 字（>12 字 warning）
- **砍虚词**：去掉 `公开/全量/细项/与分析/情报简报/深度解读` 等不增信息的限定词（命中 warning）
- **但别过度简化**：保留体现**区分价值**的词——如 `信披属实 / 需求匹配 / 海外执法 / 深挖 / 可信度`，为短而短反而丢价值

### 正例 / 反例
```
✅ 两融竞对情报（砍掉「券商两融政策…情报简报」的虚词）
✅ 管理层叙事与信披核查（保留"信披"价值，只去「财报…属实」冗余）
❌ 财报管理层叙事与信披属实核查（14 字偏长）
❌ 管理层叙事核查（过度简化，丢了"信披"这个核心价值）
```

## R3. search_plan 必须按数据源聚类 [error]

### 规则
`search_plan` 必须包含：
- ≥ 2 个 `**[[<标签>] <维度名>]**` 形式的源聚类维度
- 每个维度有 `**数据路由**` / `**执行动作**` / `**验证策略**` 三段
- 建议附 `信源偏好` / `避坑指南` / `数据裁决逻辑`（warning 级）

### 为什么
- 一次去一个数据源、把多类信息一并拉回来 → 调研效率高
- 按章节顺序线性走 → 同一数据源被反复请求，浪费 credits & 时效不一致
- "执行动作" 让 LLM 知道捞什么字段
- "验证策略" 让 LLM 知道多源冲突时怎么裁决

### 反例
- 维度数 < 2（不需要聚类）
- 维度名不带 `[[标签]]` 前缀（无法反向追溯支撑哪章）
- 维度只写"获取公司信息"无 数据路由 / 执行动作 / 验证策略

## R4. report_format 主标题必须含变量 [error]

### 规则
- 必须以 `> **关键配置**` 引用块开头
- 主标题 `# ...` 必须含三段式变量 `[属性_主体_类型]`
- 章节编号从 1 开始连续递增（1, 2, 3...）
- 章节数 ≥ 3
- 每个章节必须有 `> **[执行蓝图]**` 引用块
- 蓝图含 `**研究目标**` / `**逻辑链条**` / `**信息需求**` / `**输出形式**` 四段
- `输出形式` 不能列具体表格列名（如 `(A/B/C/D)` 形式）
- 建议末尾含标准化免责声明（warning 级）

### 为什么
- 主标题含变量 → 4 份不同主体的报告标题不同（万科 vs 海底捞），不会三份一样
- 编号连续递增 → 前端可解析目录
- 每章蓝图 → LLM 写章节时有约束，不会写空泛话
- 不列具体表格列名 → 列名应由 reporter prompt 决定，seed 只定形态

### 反例
- 主标题写死：`# 对公授信预尽调底稿`（4 个企业 4 份报告标题一模一样）
- 输出形式写：`表格 (公司名/法人/注册资本/股权)`（列名硬编码到 seed）

## R5. 任何字段不准出现工具名或决策语 [error]

### 规则
- 不出现 `get_*` / `list_*` / `find_*` / `search_tool*` / `crawl_*` 等工具名前缀
- 不出现 `建议进入 / 谨慎进入 / 暂缓进入 / 建议授信 / 拒绝授信`等决策语

### 为什么（工具名）
- 工具治理是后端事；前端 LLM 文本耦合工具名 → 工具更名/下线时全量模板要改
- 工具家族应该写"数据来源"（如"上市公司公告"），让后端路由到具体工具

### 为什么（决策语）
- 搭子是证据收集器，最终决策权在专业人员/审批委员会
- 越界为最终判决 → 法律风险(授信批复、合规判定、海外制裁判定 类似)

### 反例
- `用 get_disclosure_announcements 工具拉公告` — 写工具名
- `综合评估后给出"建议进入/谨慎进入/暂缓进入"三档结论` — 越界

## R6. 三段式变量三处一致 [error - 当任一处不一致]

### 规则
`input_form_spec` 里定义的变量、`report_format` 主标题里用的变量、`> **关键配置** · 目标对象` 里引用的变量，三处的命名**必须完全一致**。

### 为什么
- 前端有联动渲染：用户在 input_form_spec 填的值会被替换到三处
- 任一处不一致 → 用户看到"标题写 [目标_授信_企业]，但报告内容讲的是别的"

### 反例
- input_form_spec: `[目标_授信_企业]`
- 主标题: `[企业_授信_主体]` — 变量名不一致

---

## R7. 章节蓝图四件套齐全 [error]

### 规则
每个 `> **[执行蓝图]**` 引用块内必须含：
- `**研究目标**`
- `**逻辑链条**`
- `**信息需求**`
- `**输出形式**`

### 为什么
- 缺研究目标 → LLM 不知道这章在解决什么问题，写出泛泛之谈
- 缺逻辑链条 → 章节内证据散乱
- 缺信息需求 → LLM 找不到该捞什么数据
- 缺输出形式 → 章节呈现形式不可控（一会儿表格一会儿段落）

## R9. task_input 必须是代表性可执行样例 [error 可执行 / warning 代表性]

> 元字段铁律 —— 仅当模板设了 `task_input` 时生效；字段缺省允许。

### 规则
- **可执行** [error，机检]：单行；非空白；长度 ≤ 30；不含引导词（请输入 / 请提供 / 可选补充 / 可提供 / 可选填 / 默认: / 企业名称 / 公司名称 / 主体名称 / 统一社会信用代码 / 示例 / 例如 / 如: / 占位 / placeholder）；不含 `<...>` 或 `[...]` 形式占位变量
- **代表性** [warning，人工设计审视，无 validator]：在该模板擅长的场景里挑一个**信号丰度高、跨场景演示效果好的具体主体**（如 `宁德时代` 配 `对公授信预尽调底稿`），让首次试用者一次就看到搭子价值

### 为什么
- `task_input` 是前端「使用推荐方案」按钮直接 submit 的 agent 输入字符串。
  UI 在 `推荐方案 ({task_input})` 单括号内单行渲染 —— **过长 / 含引导词会把按钮挤变形 + 跑出空报告**
- placeholder 引导文案应放 `input_form_spec`（那是单行 helper），不是 `task_input`
- **代表性 ≠ 任意可执行值**：若样例主体无信号（如 `贵州茅台` 配信用尽调），跑出来空空，反伤产品价值感知

### 反例
- `请输入企业名称或统一社会信用代码。可选补充：所在行业、拟授信金额…` — 引导文案，属 `input_form_spec`
- `贵州茅台` 配 `对公授信预尽调底稿` — 形态对但代表性弱（消费白马信用信号过净）
- `恒大集团` 配 `季度财报点评` — 已退市、数据源不全，跑出来报错

### 正例
- `对公授信预尽调底稿` × `宁德时代`（产业链张力 + 海外业务 + 持续融资，演示信号丰富）
- `季度财报点评` × `比亚迪`（业务多线、披露完整、季度变化明显）
- `高风险主体公开合规快照` × <有监管处罚或跨境关联的代表性中型主体>

## R10. report_format 中"报告时间"字段名 + 格式 + 位置必须严格 [error]

### 规则

主标题正下一行必须是字面 `报告时间: YYYY年MM月DD日`(冒号后一空格)。

- 字段名**只能是 `报告时间`**:`报告生成时间` / `生成时间` / `报告日期` / `生成日期` 等变体一律**不允许**(error)。
- 占位符**写字面 `YYYY年MM月DD日`**:不要用 `<由 LLM 填充>` / `{CURRENT_DATE}` / `<...>` 等自创占位。
- 位置:**主标题(`# ...`)的下一行**,不放报告末尾,不放免责声明引用块里。
- **不带 `>` 引用块前缀**。

### 为什么

- 后端 reporter prompt(`cubemanus src/prompts/reporter.py:155/170/188` + `mimic.py:67/94`)和**前端报告渲染**都按 key `报告时间` 精确解析。字段名错,前端识别不到这行(或当成普通正文渲染),用户看不到生成时间。
- `YYYY年MM月DD日` 是 reporter LLM 识别的字面 pattern,reporter 会用后端环境变量 `{CURRENT_YEAR}/MONTH/DAY` 自动填入北京时间。换其他占位符,reporter 可能漏填,字面 `<由 LLM 填充>` 残留在交付报告里。
- 位置上,前端期望它紧跟标题——放报告末尾时部分模板会被前端裁掉或位置错位。

### 反例

- `> 报告生成时间：<由 LLM 填充>`(末尾,带引用块前缀,字段名错,占位符不规范)
- `生成日期: 2026-05-28`(字段名错,格式错)
- `报告时间: {CURRENT_DATE}`(占位符不规范,reporter 不会替换 `{CURRENT_DATE}`)
- 缺该字段(用户看不到报告时间;若该模板已上线,前端会标"未知日期")

### 正例

```markdown
# [目标_授信_企业] 对公授信预尽调底稿
报告时间: YYYY年MM月DD日

## 1. ...
```

`validate_template.py` 机检此项;`test_template.py` 跑后会再验产出报告里没有残留 `<由` 字面 + 含 `报告时间` 字段。

## 设计意图（可读 [`buddies-architecture`](https://github.com/cubemanus/buddy-architecture)）

铁律不是为了刁难，而是为了让搭子在以下场景**自动给力**：

| 场景 | 哪条铁律救你 |
|---|---|
| 用户拿到的 4 份不同主体报告标题一模一样 | R4（主标题含变量） |
| 报告里反复写"建议授信"被合规吐槽 | R2、R5 |
| 后端工具更名，所有模板都要改 | R5（无工具名硬编码） |
| 同一数据源被 LLM 重复请求浪费 credits | R3（按源聚类） |
| 用户填了输入但报告里没替换上 | R6（三处变量一致） |
| LLM 把章节写成废话流水账 | R4 + R7（蓝图四件套） |
| 报告写到一半被 hallucinate 出虚假表格列 | R4（不列具体列名） |
| 「使用推荐方案」按钮把 placeholder 文案当 query 跑出空报告 | R9（task_input 可执行） |
| 首次试用样例跑出来空空，价值感知差 | R9（task_input 代表性） |

## 设计经验法则 (Design Heuristics) — 非铁律

下面这些不是 hard rule(无 validator,不阻断 `+create`),只是 author 设计模板时的经验法则:

### H1. 章节数与维度数的"反比关系"建议

- 章节数 ≤ 15
- 调研维度数 2-5 个
- 两者比例 ≈ 3:1(每个调研维度平均支撑 3 个章节)

**为什么**:
- 章节多于 15 → 用户读不完,长报告反而稀释关键信息
- 调研维度 > 5 → LLM 注意力分散,每个维度都浅尝辄止
- 比例失衡 → 维度浪费或章节缺乏支撑

**例子(对公授信)**:
- 章节 13 个:动态识别 / 主体核验 / 行业景气 / 财务实证 ×3 / 评级 / 司法 / 处罚 / 舆情 / 自查 / 免责 ...
- 维度 4 个:[[主体核验]] / [[财务实证]] / [[行业景气]] / [[经营动态]]
- 比例 13:4 ≈ 3.25:1 ✅
