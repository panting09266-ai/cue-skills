# 用户参考材料 → 4 字段映射规则

在 `+author` 流程的 Stage 0，用户会向 agent 提供与场景定义相关的本地资料。本文规定 agent 如何**只在客户端 / agent 上下文里**消化这些材料，并把它们结构化吸收进 4 个字段。

## "材料" 的明确定义

凡符合以下之一，**默认按材料处理**（不进 payload，不上传 Cue）：

- **任何本地文件**：PDF / Word / Markdown / Excel / 截图 / 报表
- **链接背后的内容**：URL 在 agent 端 fetch 回来的正文（即使是公开网页，也按材料处理）
- **用户粘贴 ≥ 3 行连续文本**：可视为引用了某份原稿/规范
- **任何客户/项目/案件相关命名**：真实客户名、内部案号、未公开金额、内部代号
- **内部规范/SOP 段落**：监管要求、内审清单、岗位 SOP

**例外**（可写进 payload 的）：

- 用户明确说"这就是我想要的模板措辞"——例如 `goal` 字段的草稿、章节标题
- 通用名词解释、行业术语——非客户专属

判断不准时，**默认按材料处理**；让用户在 `+create` 前确认。

## 核心原则

1. **材料只在 agent 上下文里用** — 不上传到 Cue API、不写进 `create/update` payload、不入 git
2. **抽结构、不抽原文** — 起草模板时引用的是材料的"结构 / 字段 / 基调"，不是字面文本
3. **敏感字段必须脱敏** — 真实客户名、内部金额、未公开内控规则等出现在抽取出的结构里时，要替换成占位符（`[目标_授信_企业]` / `<金额>` / `<内控规则名>`）
4. **告知用户** — 完成抽取后明确说"已用作模板起草，原文未上传"

## 材料类型与提取方法

### A. 已有报告样例（PDF / Word / Markdown / 粘贴文本）

**最有价值的材料**。同事手里写过的同类报告是反推 `report_format` 章节结构最直接的来源。

提取步骤：

| 步骤 | 动作 | 落到哪个字段 |
|---|---|---|
| 1 | 抓主标题 | `title` + `report_format` 主标题（替换主体名为三段式变量） |
| 2 | 抓一级目录（章节） | `report_format` 章节列表（最多 13 节，超过要砍） |
| 3 | 抓每章关键二级结构 / 表格列名 | `[执行蓝图]` 的 `信息需求`（但不能列具体表格列名） |
| 4 | 抓文风（克制 / 攻击力 / 结论先行） | `关键配置 · 基调设定` |
| 5 | 抓引用的数据来源（监管 / 研报 / 司法等） | `search_plan` 的"数据路由"聚类 |
| 6 | 抓免责声明语 | `report_format` 末尾免责段（标准化复用） |

注意：
- 报告里如果有"作者签字 / 客户经理"等岗位字段，写到 `goal` 的"角色"部分而不是 report 里
- 报告里如果出现"建议进入 / 谨慎进入"等决策语 → **不要照抄**，按 R5 改写成"风险信号定级 / 关键发现"
- 报告里如果具体列了表格列名 → 提取章节本身的研究目标，**别把列名搬进 seed**（R4）

### B. 行业 SOP / 内部规范 / 监管文件

例如：金融监管总局监管要求 / 内审检查清单 / 行业协会调研规范。

| 步骤 | 动作 | 落到哪个字段 |
|---|---|---|
| 1 | 抓"必须包含的核查项" | `report_format` 的章节清单（每个核查项 → 1 章或合并） |
| 2 | 抓"数据来源约束" | `search_plan` 的 `信源偏好` |
| 3 | 抓"什么不能做 / 红线" | `goal` 的边界声明（"不替代..." 段） |
| 4 | 抓"判定阈值" | 不写进 seed，建议用户走 `+test` 后再让 LLM 通过实际跑出来的报告校准 |

### C. 同类搭子描述 / 第三方模板

例如：另一家平台发布的研究助手提示词 / Notion 模板 / GitHub 公开模板。

| 步骤 | 动作 | 落到哪个字段 |
|---|---|---|
| 1 | 对照本 skill 的 7 类铁律检查可移植性 | 决定能不能直接借鉴 |
| 2 | 抽"角色定义 / 价值主张" 改写 | `goal`（注意句式必须单段非编号清单 R2） |
| 3 | 抽"输出章节" | `report_format`（按本 skill 章节规范重组） |

注意：第三方模板里如果**没有按数据源聚类 search_plan**，不要把它的线性流程抄进来；按本 skill 的聚类规则重新整理。

### D. 链接（公司官网 / 监管页面 / 研报）

用 `WebFetch` 读，但**只摘取结构性信息**，不抽业务内容。例如：

- 监管披露目录页 → 提取披露文件类型清单（年报 / 临时公告 / 持股变动 / 中报 / 一季报）→ 落入 `search_plan` 的 "执行动作" 字段提示 LLM 取哪些文档
- 行业研报目录 → 提取行业分析的常见维度（市场规模 / 竞争格局 / 技术路线）→ 落入 `[[行业景气]]` 聚类的"执行动作"
- 公司官网 → **一般不直接用**，可让用户后续在 `+test` 时把官网链接作为 task input

不要把链接内容大段引用到模板里（会让 seed 文本臃肿且过期）。

### E. 用户口头描述

如果用户没给文件、只是说"我心目中这个搭子大概是..."，把口头描述当成 A 类报告样例处理：抽结构（章节构想 / 数据源 / 基调），再走 4 字段问答。

## 反映给用户的反馈格式

每抽完一类材料，agent 用以下格式回复用户：

```
我已从你提供的 <材料名> 抽取了：
- 章节骨架: <N 个章节>
- 数据源聚类: <M 个>
- 基调倾向: <一句>
- 边界声明: <一句>

材料原文未上传到 Cue。下面进入 Stage 1（场景与角色）...
```

让用户清楚知道：哪些被吸收、原文未上传、下一步走哪。

## 反面：不要这样做

- ❌ 把整份样例报告原文塞进 `goal` 或 `report_format`（模板会臃肿且不可复用）
- ❌ 把客户真实名字 / 真实金额 / 真实身份证号留在抽取结果里
- ❌ 跨用户复用从 A 用户材料抽出的结构（即使是 agent 自己记的"模板"也不行；每次 +author 现抽现用）
- ❌ 把材料上传到 Cue API 的任何 endpoint（既不到 templates，也不到 chat/stream，也不到 generate_template）

## 与铁律的关系

| 铁律 | 材料摄入时怎么对齐 |
|---|---|
| R1（input_form_spec 三段式） | 从材料抽到的主体名 → 替换成三段式变量 |
| R2（goal 单段攻击力） | 借鉴材料里的"角色 / 用户 / 痛点"语，重组成一段 |
| R3（search_plan 按数据源聚类） | 即使材料按"先看 A 再看 B"线性走，重新按"一次去一个源拉多类信息"聚类 |
| R4（report_format 主标题含变量） | 材料里的固定主标题 → 加变量 |
| R5（无工具名 / 无决策语） | 材料里出现"建议进入"等决策语 → 改写或丢弃 |
| R6（变量三处一致） | 抽完后整体复核 input_form_spec / 主标题 / 关键配置 三处 |

完成材料摄入后立刻进入 Stage 1-4 引导起草，每段起草后跑 `+validate` 反馈。
