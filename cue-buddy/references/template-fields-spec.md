# 模板字段格式详解

一个 Cue "搭子" 模板由 4 个 LLM-consumed 字段 + 3 个元字段构成。本文用业务语言、可复用片段说明每个字段。完整铁律见 [`hard-rules.md`](hard-rules.md)。

---

## 元字段（提交时必带，前端展示用）

| 字段 | 含义 | 例子 |
|---|---|---|
| `title` | 模板在工作台显示的卡片标题 | `对公授信预尽调底稿` |
| `primary_category` | 一级分类（行业域） | `银行 / 投研 / 资管 / 合规 / 监管研究` |
| `secondary_category` | 二级分类（**受控场景**） | `预尽调 / 公开合规快照 / 财报点评 / 投顾对比 / 私募尽调` |

> **secondary_category 是 playbook 的场景轴（受控词表）**：`+author` 时优先从 `+capabilities` 返回的 `playbook_scenes` 里**原样选一个**最贴合的，让同类搭子在 cuecue.cn/playbook 聚类到同一场景；确无贴合的再自拟 6-8 字。`+validate` 会对不在词表内的 secondary_category 给 warning（不阻断创建——自定义场景仅在积累足够同类搭子后才自动浮现成 playbook 场景）。词表单源在服务端，随业务演进。

| `source_conversation_id` | 模板"来自"的对话 ID（用 `seed:<slug>:v1`） | `seed:corporate-credit-pre-diligence:v1` |

## 1. `input_form_spec` — 用户输入表单规范（一行）

> "搭子描述/介绍" 这类自由文字实际由 `goal` 字段承担——它定义角色与价值，前端从中自动派生一句卡片描述。

### 作用
告诉用户在选这个搭子后**必填什么 / 可填什么 / 不填时取什么默认**。前端会把 `[...]` 部分变成可编辑输入框。

### 格式（强制）

```
需提供: [<属性>_<主体>_<类型>]，可提供: [<属性>_<主体>_<类型>] (默认: <默认值>)。
```

- **单行**，不能含换行
- 必须以 `需提供:` 开头
- 至少一个三段式变量 `[属性_主体_类型]` —— skill 会帮你命名

### 三段式变量命名规则

| 段位 | 含义 | 例子 |
|---|---|---|
| 属性 | 这个对象在你场景中的角色 | `目标` / `关注` / `对比` |
| 主体 | 这个对象是什么 | `授信` / `投研` / `合规` |
| 类型 | 数据形态 | `企业` / `主题` / `产品` |

### 示例

```
需提供: [目标_授信_企业]，可提供: [关注_风险_主题] (默认: 通用对公授信审查)。
```
```
需提供: [目标_合规_主体]，可提供: [关注_风险_主题] (默认: 全维度公开合规筛查)。
```
```
需提供: [目标_点评_公司]，可提供: [关注_季度_主题] (默认: 最近一期财报)。
```

## 2. `goal` — 搭子角色与价值主张（一段）

### 作用
告诉 LLM **你是谁、为谁服务、解决什么具体痛点、产出什么用户价值**。这一段决定整份报告的语气和重点。

> **goal 就是搭子在 playbook 卡片上的简介。** 讲**解决什么问题 / 给什么价值**，
> 不讲怎么实现。它同时也是给 researcher 的目标陈述——简洁的目标 + 详尽的 search_plan
> 比"运行手册式长 goal"跑得更好、卡片也更好看。

### 格式（强制）

- **单段**，不写编号清单
- **简洁有力**：~40-80 字；>100 字 warning、>200 字 error。维度与"怎么做"放 search_plan，别堆进 goal
- **价值优先**：以"解决什么问题/给什么价值/产出什么"开头；不要"作为…(长角色)…助手，围绕…，[一长串怎么做]"的运行手册式前缀
- **不泄漏实现**：不出现 LangGraph/循环图/批处理/MECE/pipeline/状态机/评估模型/估值模型/自适应/大模型 等内部表述
- **不硬码主体**：用占位变量或泛指（卡片会泛化显示 `[属性_主体_类型]`；硬码会出现"作为 中信证券 …助手"错位）
- **不写免责**：`不替代/不构成/仅做…/严格不…` 放 search_plan 避坑指南或 report_format

### 价值优先句式（参考，不强制套用）

```
<动词：核查/比对/穿透/筛选/挖掘…> <对象与关键维度（简）>，<识别/研判/产出 …价值结论>。
```

### 示例

```
✅ 横向比对各主要券商的两融政策（折算率、标的、保证金、集中度），识别竞对抢客与风险收缩
   信号、研判本方杠杆吸引力强弱，产出晨会可直接用的竞对情报。   (73字)
✅ 多源穿透授信企业的主体真实性、股权关联、经营持续性与合规风险，自动补全公开披露盲区，
   产出可上初审会、带证据链的预尽调底稿。   (取代旧的 330+ 字运行手册式 goal)
```

### 反面：错误写法

❌ 编号清单：
```
1. 提供企业基本信息
2. 提供财务数据
3. 提供风险评估
```
（无角色、无场景、无攻击力）

❌ 决策语：
```
判断 [目标_授信_企业] 是否建议进入授信审查
```
（搭子是证据收集器，不是决策者，禁用"建议进入/谨慎进入/暂缓进入"）

## 3. `search_plan` — 调研策略（按数据来源聚类）

### 作用
告诉 LLM **去哪些数据来源取证、怎么取、怎么相互验证**。这一段决定调研深度和数据可信度。

### 核心理念：按数据源聚类，不是按章节顺序线性走

传统模板按"先做基本面 → 再做财务 → 再做风险"线性写。Cue 不这么做。Cue 按"一次去一个数据来源把多类信息拉回来"聚类，每个维度可以同时支撑报告的多个章节。

### 格式（强制）

```markdown
**[[<聚类标签>] <维度名>]**
- **数据路由**: <用什么工具家族 / 什么 API / 什么文档源；不写具体工具名>
- **执行动作**: <这个维度去捞什么字段 / 看什么报告 / 验证什么口径>
- **验证策略**: <多源冲突时怎么裁决 / 数据缺失时怎么标注 / 时效性怎么判>
```

聚类标签是给报告章节做反向追溯用的（比如 `[[主体核验]]` 支撑章节 2/3，`[[财务实证]]` 支撑章节 4/5/6）。

### 维度数量建议

- **下限 2 个**（少于 2 个就不需要聚类，直接写一段就行）
- **上限 5 个**（多于 5 个会让 LLM 失焦，调研失效）
- 推荐 3-4 个

### 还应该包括（建议但非强制）

- **信源偏好**: 哪个信息源优先（监管披露 > 行业研报 > 媒体）
- **避坑指南**: 什么数据容易误读（如 6 位数字可能是公司代码也可能是金额）
- **数据裁决逻辑**: 多源冲突的优先级，如"以最新一期审计报告为准"

### 示例（对公授信，简化版）

```markdown
**[[主体核验] 公开身份信息]**
- **数据路由**: 工商登记 / 上市公司公告 / 港股披露 / 海外母公司 SEC 文件 / 非上市公司监管 ID
- **执行动作**: 法定代表人、注册资本、股权穿透至自然人、上市状态、最新一期年报披露日
- **验证策略**: 工商与披露不一致时,以披露为准并标注差异;未披露字段标"待补充"

**[[财务实证] 三年财务与审计意见]**
- **数据路由**: 上市公司年报现金流表/资产负债表/利润表 / 港股核数师报告 / 评级机构观点
- **执行动作**: 拉取近三个会计年度核心指标(营收/净利润/经营性现金流/总资产/负债率),
  核对审计意见(无保留/保留/否定/无法表示);非上市企业用纳税信用与行业基准估算区间
- **验证策略**: 财报与评级口径有冲突时,以评级机构最新报告为准并写"评级窗口";
  财报缺失时不编造,改用同行业可比数据 + 明确标注"行业基准估算"

**[[行业景气] 行业风向与监管环境]**
- **数据路由**: 行业协会数据 / 证监会/金融监管总局监管公告 / 海外监管(SEC/EU 等)
- **执行动作**: 行业增长 / 政策方向 / 海外制裁清单 / 同业评级窗口
- **验证策略**: 行业数据用近 12 个月口径;制裁清单以官方文件为最终依据

**[[经营动态] 司法/舆情/合规]**
- **数据路由**: 司法公开 / 舆情聚合 / 监管处罚 / 行业新闻
- **执行动作**: 近 24 个月被执行/重大诉讼/行政处罚/突发负面事件
- **验证策略**: 媒体二次转载内容必须找到一手出处;无法定位一手的标"待补充"
```

### 反面：错误写法

❌ 章节顺序线性走（"先做章节 1，再做章节 2"）：违反核心理念
❌ 写工具名（`get_disclosure_announcements`）：违反铁律 5
❌ 写"建议进入授信审查"等决策语

## 4. `report_format` — 报告交付形态

### 作用
告诉 LLM **报告交付给用户时长什么样**。这是用户拿到的最终产物，决定可读性、可签字性、可回查性。

### 格式（强制）

#### 4.1 顶部"关键配置"引用块（强制开头）

```markdown
> **关键配置**
> - **目标对象**: [<引用 input_form_spec 里的三段式变量>]
> - **报告类型**: <"底稿" / "速递" / "点评" / "深度">
> - **基调设定**: <"客观克制" / "结论先行" / "对比分析">
> - **核心命题**: <一句话回答"本搭子为什么存在">
```

#### 4.2 主标题（强制含变量）

```markdown
# [<同 input_form_spec 三段式变量>] <场景>底稿
```

例如：
- `# [目标_授信_企业] 对公授信预尽调底稿`
- `# [目标_合规_主体] 主体合规风险公开快照底稿`

主标题不写死字符串，让 LLM 用具体主体名替换变量。

#### 4.3 各章节（每章带"执行蓝图"）

```markdown
## <章节号>. <章节标题>

> **[执行蓝图]**
> - **研究目标**: <一句话讲这章解决什么问题>
> - **逻辑链条**: <这章证据如何串起来支撑结论>
> - **信息需求**: <这章必须包含什么字段 / 子段>
> - **输出形式**: <叙述段落 / 表格 / 列表;但不要列具体表格列名>

<空行,后面让 LLM 按蓝图填充>
```

##### 章节编号：从 1 开始连续递增

##### "输出形式"：禁止列具体表格列名

❌ 错：`输出形式: 表格 (公司名/法人/注册资本/股权/上市状态)`
✅ 对：`输出形式: 主体身份核验表 + 关键差异说明段落`

具体列名应由后端 reporter prompt 决定，不是 seed template 的事。

#### 4.4 报告时间字段(**强制**，前端依赖)

主标题正下一行必须含 `报告时间: YYYY年MM月DD日` 字面占位:

```markdown
# [目标_<场景>_主体] <场景>底稿
报告时间: YYYY年MM月DD日

## 1. ...
```

**铁律(违反会影响前端渲染):**
- 字段名**只能是 `报告时间`**——不是 `报告生成时间`、不是 `生成时间`、不是 `报告日期`。后端 reporter prompt(`cubemanus src/prompts/reporter.py:155/170/188` + `mimic.py:67/94`)和前端解析都钉死了这个 key。
- 格式**写字面 `YYYY年MM月DD日`**(不要用 `<由 LLM 填充>` 或 `{CURRENT_DATE}` 等自创占位符)。reporter LLM 识别这个 pattern 并填入当前北京时间。
- 位置:**主标题的下一行**(不是报告末尾)。
- 不带 `>` 引用块前缀,也不放进免责声明块。

#### 4.5 标准化免责声明（强烈建议）

最后一节加：

```markdown
---

> 本报告基于公开信息编制，不构成任何投资建议、授信决策或法律意见。
> 数据时效性以披露日为准；多源冲突已按 search_plan 中的裁决逻辑处理。
```

### 示例：3 个简单章节

```markdown
> **关键配置**
> - **目标对象**: [目标_点评_公司]
> - **报告类型**: 季度财报点评
> - **基调设定**: 客观克制
> - **核心命题**: 用一份不超过 8 分钟可读的笔记,帮投研助理快速建立对本季业绩的事实判断

# [目标_点评_公司] 季度财报点评
报告时间: YYYY年MM月DD日

## 1. 季度核心数据

> **[执行蓝图]**
> - **研究目标**: 提取本季最关键的 3-5 个财务指标变动
> - **逻辑链条**: 收入 → 毛利 → 净利 → 经营性现金流
> - **信息需求**: 营收 / 同比 / 毛利率 / 净利润 / 经营性现金流;均给同比和环比
> - **输出形式**: 关键指标卡 + 简要点评段落

## 2. 业务动能拆解

> **[执行蓝图]**
> - **研究目标**: 拆解收入按业务线 / 区域 / 产品
> - **逻辑链条**: 按管理层 KPI 维度拆分,找出本季最强与最弱业务
> - **信息需求**: 三个维度的拆解口径
> - **输出形式**: 业务拆解段落,标注关键拐点

## 3. 风险信号与不确定性

> **[执行蓝图]**
> - **研究目标**: 列出本季报告与上季管理层指引的偏离 / 新增风险
> - **逻辑链条**: 管理层指引 → 实际数据 → 偏差归因
> - **信息需求**: 偏离条目 / 一次性影响 / 管理层解释
> - **输出形式**: 偏差列表 + 自查不确定性段落

---

> 本报告基于公开信息编制,不构成任何投资建议。
```

## 三段式变量的"连贯性"

`input_form_spec` / 主标题 / "关键配置·目标对象" 三处的三段式变量**必须完全一致**。前端会用它做联动渲染：用户在 input_form_spec 填的值，会自动替换到标题与配置块。

不一致会导致用户看到"标题里是 [目标_授信_企业]，但实际内容讲的是别的"。

## 5. 任务调度元字段（可选）

模板还可以附带"推荐运行方案"——前端在工作台展示模板时，会渲染一个"使用推荐方案"按钮，让用户一键以预设值跑一次。三个相关字段：

### 5.1 `task_input` — 推荐方案的具体执行值

#### 作用
用户点 TaskManagerDialog「使用推荐方案 ({task_input})」按钮时，前端**直接把 `task_input` 字符串作为 agent 输入提交**（即 `input_content`）。同时也作为按钮内括号里的提示渲染。

#### 契约（强制 + 强烈建议）
- **必须是「可直接执行的具体值」** —— 跟用户在 TaskSchedulerDialog 自己填的内容同语义（如企业名、主体名、产品名）
- **单行**；建议 **≤ 30 字符**（前端单括号渲染，过长 UI 挤变形）
- **不得含引导词**：`请输入` / `请提供` / `可选补充` / `可提供` / `可选填` / `默认:` 等 → 这些是 `input_form_spec` 的活
- **应是代表性主体**（设计审视，无 validator）：在模板擅长场景里挑一个"信号丰度高、跨场景演示效果好"的具体主体，让首次试用者跑一次就看到搭子价值

#### 反例

```python
# ❌ 引导文案塞进 task_input
"task_input": "请输入企业名称或统一社会信用代码。可选补充：所在行业、拟授信金额、贷款用途..."
# 这是 input_form_spec 该装的内容；UI 「推荐方案 ({task_input})」按钮挤变形 +
# 按钮把整段说明当 agent 输入跑，必空报告

# ❌ 形态对，但代表性弱
"task_input": "贵州茅台"  # 对公授信预尽调：消费白马信用信号过净 → 首次试用空看
"task_input": "恒大集团"  # 季度财报点评：已退市数据源不全 → 跑出来报错
```

#### 正例

```python
# ✅ 对公授信预尽调
"task_input": "宁德时代"  # 产业链张力 + 海外业务 + 持续融资 → 演示信号丰富
# ✅ 季度财报点评
"task_input": "比亚迪"    # 业务多线 + 披露完整 + 季度变化明显
```

> 主体合规风险公开快照 等敏感场景的 `task_input` 主体由 buddy owner 在
> seed 阶段直接选定一个**经授权、可公开检索**的具体主体 string,不要写
> `<有...的代表性主体>` 这类占位 — 占位会被 R9 lint 直接拦截,并且会把
> 首次试用者引向"按 placeholder 填一个意思相近的字符串"的错误模式,
> 复刻引发本规则的原 bug。

### 5.2 `schedules` — 推荐运行节律（可选，list[dict]）

公开 API 字段。前端在「推荐运行」UI 把 schedules 渲染为预设运行时点 chip。

```python
"schedules": [
    {"type": "daily",  "time": "09:00", "date_param": None, "dates": None},
    {"type": "weekly", "time": "09:00", "date_param": "MON", "dates": None},
]
```

字段:
- `type`: `daily` / `weekly` / `monthly` / `once`
- `time`: `HH:MM` 24 小时制
- `date_param`: 周字段(`MON` / `TUE` / ...) 或单次日期(`2026-12-31`),按 `type` 用
- `dates`: 月内多日(如 `["1", "15"]`),按 `type=monthly` 用

约束:
- 后端 schema `ScheduleConfig.type` / `.time` 必填字符串;`date_param` / `dates` 可空
- `schedules` 整体可空(省略=不设推荐节律 / partial update=保留原节律)
- 传时至少 1 条(`min_length=1`,空数组 422)

不写就是"无推荐运行节律，仅作为按需调用的模板"。

### 5.3 Legacy: `task_cron_expressions` / `task_configs` (server-internal, 勿外发)

这两是 **DB 列名**,不是公开 API 字段。历史上 seed 脚本直写 DB(`INSERT INTO template ... task_cron_expressions, task_configs`),所以你在 prod seed 代码里会见到。但通过 `cue_api.create_template()` / `update_template()` 调时:

- 把节律写在 `schedules` 字段就够了
- 若你拿到的 legacy payload 里只有 `task_configs`(没有 `schedules`),`cue_api._normalize_template_payload()` 会自动派生 `schedules`(`name` 字段会被丢)
- `cue_api` 始终 strip outbound 的 `task_cron_expressions` / `task_configs`(server 默认 `extra='ignore'` 会静默吞,我们提前拦)
- 直接传 `cron_expressions`(拼错 — 缺 `task_` 前缀)会本地 raise

> R9 铁律见 [`hard-rules.md`](hard-rules.md#r9-task_input-必须是代表性可执行样例-warning)。`schedules` 由后端 Pydantic schema 强校验,写错会 422;legacy 字段当前无机检,写在 outbound payload 等于无操作。
